# Plugins may contribute build-time tools

Status: accepted

A plugin can now ship a **build tool**: an executable compiled for the build
host, wired by `generate()` as a named top-level build step, with comptime
access to the consumer's generated command registry — and never linked into
the shipped binary. The first user is `zcli_docs`, a **build-only** plugin
that replaces the `generateDocs()`/`DocsConfig` core build API (removed, no
compatibility shim). Doc generation is author-time work; it has no business
occupying bytes in every user's binary.

## Context

The documentation generator was the framework's one hardcoded build-time
consumer of command metadata: `generateDocs()` built a host executable from
`doc_gen_main.zig` inside core, imported the registry into it, and registered
a `docs` step. The mechanism was right — a host tool over comptime registry
data is Zig's idiom for build-time generation (comptime proper cannot write
files) — but it was a one-off carved into core's build API rather than a
capability of the plugin system. Meanwhile ADR-0027 already establishes that
plugins are compile-time participants in the registry; runtime hooks and
commands were simply the only surface they had.

Extracting doc generation as a *runtime* `docs` command (the completions
plugin's shape) was considered and rejected: completions belong in the binary
because end users generate them on their machines at install time; docs are
generated by the CLI's author at build time. The extraction therefore required
generalizing the build-time surface, not bending doc generation into the
runtime one.

## The contract

`PluginConfig` gains two fields:

- `tool: ?ToolConfig` — `{ step, description }`. When set, `generate()`
  compiles the plugin's tool source for `b.graph.host`, gives its root module
  three imports (`command_registry`, `zcli`, `tool_config`), and registers
  `zig build <step>`. Tool steps stay off the default install/test path.
- `build_only: bool` — the plugin ships no runtime module: nothing is
  registered in the generated registry, nothing is linked into the binary,
  and its `.link` hook (if any) is not applied.

Per registration model:

- **Built-in:** the `Builtin` table declares the tool (source is `tool.zig`
  next to the plugin's source) and build-only-ness; `builtin(.docs, .{...})`
  fills everything in. Step names are fixed per plugin.
- **External package:** `.tool = .{ ... }` on the registration is the
  **explicit opt-in**, and the package exposes its tool's root source as a
  module named `tool` (parallel to the `plugin` module contract; declared
  without a target — the consumer resolves it to the build host). Explicit
  beats auto-detection here for a structural reason: the build system cannot
  introspect the plugin's source, so with auto-detection there is nowhere for
  the step name to live. A package with no `plugin` module is inferred
  build-only; registering one without `.tool` is a loud config error.
- **Local `plugins_dir` plugins:** not yet supported; the natural extension is
  ADR-0006-style convention discovery of `plugins/<name>/tool.zig`, deferred
  until something needs it.

Step-name collisions (two plugin tools, or a step the project registered
before `generate()`) are caught up front and stop the build with a named
diagnostic, mirroring the plugin-identifier collision checks.

## Config flows as a rendered literal with a typed landing site

Plugin config is written as an anonymous struct in build.zig but consumed in a
different compilation (the generated registry, or the tool), so the only
transport is generated source text. The previous mechanism rendered the config
as a pre-formed `".init(.{...})"` call string on the raw-source field
`PluginConfig.init` — which both baked in one destination and invited
hand-written code strings into generated source.

Now the renderer produces a **bare struct literal**, held on
`PluginConfig.config` and filled only via the public `zcli.config(.{...})`
helper (`builtin()` calls it internally). Code generation chooses the
destination, and each destination gives the literal a result type — which is
what makes untyped literals like `.verification = .{ .minisign = "KEY" }`
(a union payload) or `.formats = &.{ "markdown", "man" }` (a tuple pointer
coercing to a slice) evaluate correctly:

- runtime plugin: `@import("name").init(<literal>)` — typed by `init`'s
  `Config` parameter;
- tool: a generated `tool_config` module containing
  `pub fn config(comptime Config: type) Config { return <literal>; }` —
  the tool calls it with its own `Config` type.

## Consequences

- **Breaking:** `generateDocs()` and `DocsConfig` are gone. Migration is one
  line: `zcli.builtin(.docs, .{ .formats = ..., .output_dir = ... })` in the
  `generate()` plugins list. Output is byte-identical; the step keeps its name.
- The man-page `.TH` date is computed when the tool runs (honoring
  `SOURCE_DATE_EPOCH`) instead of being injected as a build option — the tool
  runs at build time, so its wall clock *is* the build date, and no date is
  baked into any compiled artifact (the old scheme existed to avoid a daily
  registry cache bust; now nothing needs avoiding).
- The registry surface a tool consumes (`command_info`,
  `global_options_info`, `app_name`/`app_version`/`app_description`) is the
  same projection completions and help already rely on — plugin tools add a
  consumer, not a new contract.
- Future build-time work (pre-generating completion scripts for packaging,
  command-tree JSON export, metadata lint gates) now has a home that isn't a
  new core build function each time.
